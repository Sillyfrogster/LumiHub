# LumiHub Bug Audit (2026-04-21)

## 1) Leaderboard creators route points to the wrong user identifier (requested leaderboard bug)
- **Severity:** High
- **Why this is a bug:** The leaderboard creator rows link to `/user/${c.userId}` in the frontend, but the app’s public profile route is `/user/:discordId` and backend lookup is by `discord_id`.
- **Impact:** Clicking a creator in the leaderboard can route to a non-existent user page (404) when `userId` is an internal UUID.
- **Evidence:**
  - Frontend link uses `userId`: `src/pages/leaderboard/Leaderboard.tsx`
  - Profile route expects `:discordId`: `src/App.tsx`
  - Backend profile endpoint resolves `discord_id`: `backend/src/routes/user.routes.ts`
- **Fix direction:** Return `discordId` in creator leaderboard API and link with that value.

## 2) Leaderboard creator avatars are built with internal UUIDs instead of Discord IDs
- **Severity:** High
- **Why this is a bug:** Discord avatar URLs require Discord user IDs, but the frontend builds CDN URLs from `c.userId` (internal DB UUID in backend query).
- **Impact:** Broken avatar images across creator leaderboard rows.
- **Evidence:**
  - Avatar URL uses `c.userId` in `src/pages/leaderboard/Leaderboard.tsx`
  - Backend creator query selects `u.id AS "userId"` in `backend/src/services/leaderboard.service.ts`
- **Fix direction:** Include and use `u.discord_id` in API response for avatar/profile links.

## 3) Leaderboard API does not validate `limit` as a finite integer
- **Severity:** High
- **Why this is a bug:** `limit` is parsed with `Number(rawLimit)` without validating numeric shape. Inputs like `?limit=abc` produce `NaN`, which reaches SQL parameter binding.
- **Impact:** Can trigger 500 errors instead of a clean 400 Bad Request; also allows malformed traffic to destabilize endpoint behavior.
- **Evidence:**
  - `const limit = rawLimit ? Number(rawLimit) : undefined;` in `backend/src/routes/leaderboard.routes.ts`
  - Service clamps via `Math.min(Math.max(params.limit ?? 25, 1), 100)` which does not recover from `NaN`: `backend/src/services/leaderboard.service.ts`
- **Fix direction:** Explicitly reject non-finite/non-integer values in route layer and return 400.

## 4) OAuth `return_to` validation rejects valid cross-origin frontend URLs
- **Severity:** High
- **Why this is a bug:** `normalizeReturnTo` compares `target.origin` with `new URL(requestUrl).origin` (backend origin), but deployments often separate frontend and backend domains.
- **Impact:** Legitimate post-login redirects fail in multi-origin setups; users are forced back to `/`.
- **Evidence:**
  - `allowedOrigin = new URL(requestUrl).origin` in `backend/src/routes/auth.routes.ts`
- **Fix direction:** Validate against explicit allowlist (e.g., `env.FRONTEND_URL`) and optionally same-origin as fallback.

## 5) Chub worldbook favorites are dropped during normalization
- **Severity:** Medium-High
- **Why this is a bug:** `fromChubLorebook` hardcodes `favorites: 0` despite `ChubWorldBook` carrying a real `favorites` value.
- **Impact:** UI shows incorrect popularity signals and breaks any favorite-based ranking/filtering logic for Chub lorebooks.
- **Evidence:**
  - Source model includes `favorites`: `src/types/worldbook.ts` (`ChubWorldBook`)
  - Mapping hardcodes `favorites: 0`: `src/types/worldbook.ts` (`fromChubLorebook`)
- **Fix direction:** Map `favorites: lb.favorites`.

## 6) Chub worldbook “downloads” field is incorrectly sourced from stars
- **Severity:** Medium
- **Why this is a bug:** `fromChubLorebook` maps `downloads` to `lb.starCount`, which is a different metric.
- **Impact:** Misleading stats labels in detail/list UIs; users see “downloads” that are actually stars.
- **Evidence:**
  - `downloads: lb.starCount` in `src/types/worldbook.ts`
- **Fix direction:** Either rename displayed metric to stars for Chub lorebooks or introduce a distinct field and UI label.
